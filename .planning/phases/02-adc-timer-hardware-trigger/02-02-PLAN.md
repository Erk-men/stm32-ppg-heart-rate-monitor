Design rationale (from RESEARCH.md Section 5):
- "adc: NNNN\r\n" is approximately 10 characters (worst case 4095 + prefix = 10 chars).
- At 115200 baud with 10 bits/char: 115200 / 10 = 11520 chars/sec capacity.
- 100 Hz × 10 chars = 1000 chars/sec = 8.7% of baud capacity.
- Polling TX completes in ~870 µs per line. TIM3 period = 10 ms. No overrun risk.
- Total CPU% per sample: ~8.7%. ADC ISR adds < 1%. SysTick: < 0.1%. Well within budget.

Do NOT use printf, sprintf, or any stdlib formatting. The existing uart_write_str + uart_write_u32 API handles all output.

Do NOT remove the Phase 1 millis loop — REPLACE the entire while(1) with the ADC-polling loop above. The millis print loop is no longer needed after Phase 2.

  <how-to-verify>
Step 1 — Flash the firmware:
  In STM32CubeIDE: Run → Debug (or Run → Run) to flash via ST-Link.
  The board should reset and start running immediately.

Step 2 — Open serial terminal:
  Open any serial terminal (CubeIDE's built-in console, PuTTY, or screen) at 115200 baud 8N1 on the ST-Link virtual COM port.
  You should see the banner: "--- HeartRateSensor Phase 2 ---" followed by continuous "adc: NNNN" lines.

Step 3 — Verify output rate (100 Hz):
  Count the number of "adc:" lines printed in exactly 10 seconds.
  Expected: approximately 1000 lines (100 Hz × 10 s).
  Accept: 980–1020 lines (±2%). Fewer than 980 or more than 1020 indicates a TIM3 PSC/ARR error.

Step 4 — Verify 12-bit range (high end):
  Connect a jumper wire from the 3.3V pin on the Nucleo board (CN7 pin 16 or CN10 pin 8) to PA0 (CN10 pin 29).
  The stream should show values of 4080–4095 (within ~15 counts of full scale).
  Remove the jumper after verifying.

Step 5 — Verify 12-bit range (low end):
  Connect PA0 (CN10 pin 29) to GND (CN10 pin 20).
  The stream should show values of 0–15 (within 15 counts of zero).
  Remove the jumper after verifying.
  (Leave PA0 unconnected/floating for normal operation — it will show a noisy mid-range value, which is expected.)

Step 6 — Verify SysTick not disrupted:
  Optionally re-flash Phase 1 firmware and confirm millis() still reports 1000 ms/s. Or confirm that ADC ISR priority 1 > SysTick priority 0 is correct (priority 0 = highest on M0, so SysTick will always preempt ADC1_IRQHandler if both fire simultaneously — millis() remains accurate).

Common failure modes and diagnosis:
  - No output at all → adc_init() hanging in a spin loop (ADCAL or HSI14RDY or ADRDY wait). Check register sequence order.
  - Output rate ~11 Hz instead of 100 Hz → PSC=719 (F103 value) used instead of PSC=79. Fix PSC.
  - All values stuck at 0 → EOC not firing; EXTSEL may be wrong (EXTSEL=010 selects absent TIM2). Confirm EXTSEL_0|EXTSEL_1.
  - All values stuck at 4095 → PA0 left floating near 3.3V rail; expected if no hardware connected.
  - ISR firing at wildly high rate → EOC not being cleared (reading DR should clear it; confirm adc.c reads DR).
  </how-to-verify>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| ISR context → main() context | g_adc_ready and g_adc_sample written by ADC1_IRQHandler, read by main() — volatile qualifier is the only synchronization mechanism |
| USART2 TX → host terminal | Raw 12-bit integer values transmitted as ASCII decimal; no authentication or integrity check |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-07 | Tampering | g_adc_ready flag race | accept | M0 byte write is atomic; g_adc_ready is uint8_t; torn write impossible; worst case is one missed sample (g_adc_ready cleared just after ISR sets it) — acceptable for debug streaming |
| T-02-08 | Denial of Service | USART TX blocks while ISR fires | accept | Worst case TX duration ~870 µs; TIM3 period 10 ms; ADC ISR preempts main() immediately after TX byte completes (ISR priority 1 < SysTick 0 but both preempt main's thread); no sample is permanently lost — OVRMOD overwrites stale DR |
| T-02-09 | Information Disclosure | Raw ADC values visible on serial | accept | Debug-only output; Phase 4 will gate output behind CALIBRATION_MODE / DEBUG_VERBOSE flags; no PII or security-sensitive data at this stage |
| T-02-10 | Spoofing | USART RX not implemented | accept | No RX parsing in this phase; no attack surface from host |
</threat_model>
